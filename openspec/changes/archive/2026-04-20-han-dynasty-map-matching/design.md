## Context

华夏足迹是一个全栈项目，目标是将中国历史地理与现代旅游深度结合。系统由 Flutter 移动端 + Python 后端组成。

- **前端（Flutter）**：地图展示、浏览搜索、用户交互、离线数据缓存
- **后端（FastAPI + LangChain）**：AI Agent 编排、图像生成、行程规划、撮合服务、地理围栏

首期 MVP 聚焦汉朝（202BC-220AD），同时搭建好后端 AI 基础架构，为后续朝代扩展和 AI 功能迭代打基础。

技术约束：
- 跨平台移动端（iOS + Android）
- 后端需要支持 GPU 推理（AI 换装场景生成）
- 离线优先（核心地图功能不依赖网络）
- 可扩展：后续需支持唐、宋、明、清等朝代

## Goals / Non-Goals

**Goals:**
- 搭建 Flutter 三层架构（data / domain / presentation）
- 搭建 Python 后端 LangChain Agent 架构（FastAPI + PostgreSQL + Redis）
- 实现 100-200 条汉代地名种子数据的离线存储
- 在高德地图上分级展示汉代地名标记（州/郡/县）
- 实现本地优先 + 后端 AI 兜底的古今地名匹配引擎
- 实现历史路线规划 Agent（给定人物/朝代，生成旅行路线）
- 实现地理围栏通知（进入历史地点境内时推送）
- 实现服装/摄影撮合系统（匹配当地商家）
- 实现 AI 换装与历史场景生成（用户拍照→AI 换装→合成历史场景）
- 实现动态行程规划 Agent（预算/时间/偏好→酒店+门票推荐）
- 实现行程总结生成（照片+足迹→页面/视频）
- 打造古典中国美学 UI

**Non-Goals:**
- 用户系统/登录注册（后期再加）
- 其他朝代支持（唐宋明清等后续迭代）
- 社交分享功能（后期功能）
- Web 端（先做移动端）

## Decisions

### 前端架构决策

#### 1. 状态管理：Riverpod 3.0
**选择**: Riverpod 3.0 + riverpod_annotation 代码生成
**原因**: 编译时安全检查、天然支持 Drift 的 `.watch()` 响应式流、相比 BLoC 样板代码更少、社区主流选择
**备选**: BLoC（样板多）、GetX（类型不安全）

#### 2. 数据库：Drift (SQLite)
**选择**: Drift 2.x
**原因**: 类型安全的 SQL 构建、响应式 `.watch()` 查询、内置迁移支持、Freezed 兼容
**备选**: sqflite（无类型安全、无响应式）、Isar（过度设计）

#### 3. 地图插件：amap_map (kuloud)
**选择**: amap_map 1.x
**原因**: 最活跃维护的高德 Flutter 插件、封装原生 SDK 性能好、支持自定义标记
**备选**: amap_flutter_location（仅定位）、flutter_amap（停止维护）

#### 4. 地图标记性能策略：缩放级别过滤 + 视口过滤
**选择**: 两级过滤而非聚合
**原因**: 100-200 标记量级下足够流畅，实现简单
**备选**: 原生聚合（实现复杂）

#### 5. 路由：GoRouter
**选择**: GoRouter 14.x
**原因**: 声明式路由、深链接支持、与 Riverpod 配合好
**备选**: Navigator 2.0（底层 API）

#### 6. 种子数据：JSON 文件 + 首次启动导入
**选择**: assets/data/han_dynasty_locations.json
**原因**: 版本可控、离线可用、支持增量更新
**备选**: 远程 API 拉取（需要网络）

#### 7. 项目架构：三层 + 功能模块
**选择**: data（数据库/网络/仓库）→ domain（实体/用例）→ presentation（provider/页面/组件）
**原因**: 清晰的依赖方向、新增朝代只需加种子数据不改架构
**备选**: feature-first（跨层边界模糊）

### 后端架构决策

#### 8. AI 编排框架：LangChain (Python)
**选择**: LangChain + LangGraph
**原因**:
- Python 生态与 Stable Diffusion/IP-Adapter/ControlNet 天然集成，AI 换装（需求 4）是技术最难的点
- LangGraph 支持有状态的 Agent 工作流，适合动态行程规划（需求 5）
- Agent + Tool 体系成熟，方便集成酒店/门票/地图等外部 API
- Python 视频处理库（moviepy, ffmpeg-python）丰富，支撑行程总结（需求 6）
**备选**: Genkit（TypeScript，图像生成生态缺失）、自建（重复造轮子）

#### 9. Web 框架：FastAPI
**选择**: FastAPI + Uvicorn
**原因**: 异步原生、自动生成 OpenAPI 文档、与 LangChain 集成良好、Python 生态
**备选**: Django（过重）、Flask（异步支持弱）

#### 10. 数据库：PostgreSQL + Redis
**选择**: PostgreSQL（主存储）+ Redis（缓存/消息队列/地理位置）
**原因**:
- PostgreSQL PostGIS 扩展支持地理围栏查询（需求 2）
- Redis GEO 命令支持高效的地理围栏判定
- Redis 作为 LangChain 的对话记忆缓存和任务队列
**备选**: MySQL（无 PostGIS）、MongoDB（无地理扩展）

#### 11. AI 换装技术栈：Stable Diffusion + IP-Adapter + ControlNet
**选择**: Stable Diffusion XL + IP-Adapter (FaceID) + ControlNet (OpenPose)
**原因**:
- IP-Adapter 保证用户面部一致性（用户拍照后仍像自己）
- ControlNet OpenPose 保持姿势不变，只替换服装和背景
- 开源可控，部署在自己 GPU 服务器上，无第三方 API 调用费用
**备选**: DALL-E API（不可控、无面部一致性）、Midjourney（无 API）

#### 12. LLM 提供商：抽象层 + 多模型支持
**选择**: LangChain 抽象层，支持 DeepSeek / Claude / 通义千问 热切换
**原因**: 不同 Agent 可用不同模型（路线规划用强推理模型，简单匹配用便宜模型）
**备选**: 绑定单一 LLM（不灵活）

#### 13. 图像/视频存储：MinIO 或阿里云 OSS
**选择**: MinIO（自建）/ 阿里云 OSS（云端）
**原因**: 用户上传照片和生成的 AI 图像/视频需要对象存储，支持 CDN 加速
**备选**: 本地文件系统（不可扩展）

#### 14. 地理围栏方案：Redis GEO + 高德围栏 API
**选择**: Redis GEO 命令存储历史地点坐标，客户端上报位置后后端判定是否进入围栏
**原因**: Redis GEO 高效（O(log N)）、支持半径查询
**备选**: 纯客户端围栏（精度差、不跨设备）

## Risks / Trade-offs

### 前端风险
- **[高德地图 Key 配置]** → 用户需自行申请高德开发者 Key。Mitigation：提供详细配置文档。
- **[种子数据准确性]** → 手工整理 200 条数据可能有坐标偏差。Mitigation：优先使用考古确认的遗址坐标。
- **[amap_map 插件成熟度]** → 第三方维护。Mitigation：锁定版本号。
- **[Drift 代码生成]** → build_runner 增加构建时间。Mitigation：增量构建。

### 后端风险
- **[AI 换装 GPU 成本]** → Stable Diffusion 推理需要 GPU 服务器，单次推理约 5-10s。Mitigation：使用 SDXL Turbo 加速、请求队列削峰、初期可用云 GPU 按量计费。
- **[AI 换装效果质量]** → 服装替换和场景合成效果不稳定，可能出现伪影。Mitigation：迭代优化 prompt 和 ControlNet 参数、建立效果评估流程。
- **[LLM API 费用]** → 多 Agent 并发调用可能产生较高费用。Mitigation：路由规划用强模型、简单任务用小模型、结果缓存。
- **[酒店/门票 API 依赖]** → OTA 平台 API 可能有调用限制和佣金。Mitigation：初期可对接携程/美团开放平台、后续谈直连。
- **[视频生成耗时]** → 行程总结视频生成可能需要分钟级处理。Mitigation：异步任务 + WebSocket 推送进度、后台生成完成后通知用户。
- **[撮合系统冷启动]** → 初期商家数据不足。Mitigation：先接入第三方平台（美团/大众点评），积累用户后再建自有商家体系。
